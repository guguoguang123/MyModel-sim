# DRGate-Sim

This project generates workload-response behavior labels for DRGate. The labels
are designed to supervise a behavior-response embedding `h_b` that is distinct
from a static functional embedding `h_f`: `h_b` should learn how a node responds
across a set of workload probes, instead of predicting logic probability or
toggle probability under one single workload.

The current official label file is:

```text
simProject/outputs/behavior_labels.npz
simProject/outputs/label_meta.json
```

It contains 10,182 circuits, 8 workload probes, 4,096 vectors per probe, and
16 stochastic realizations for each stochastic probe.

## Simulation

The labels are generated by AIG-level stable Boolean simulation over
`RelatedData/graphs.npz`. DeepGate2 `labels.npz` is not used to create these
behavior labels.

The graph is interpreted as:

```text
PI_GATE  = 0
AND_GATE = 1
NOT_GATE = 2
edge_index: fanin -> gate
```

Simulation follows the forward topological level: PI nodes take sampled input
vectors, AND nodes compute the conjunction of fanins, and NOT nodes invert their
single fanin. This is not timing-aware, glitch-aware, SDF, or physical power
simulation.

## Label Definition

For each node `v` and workload probe `D_k`, the simulator first estimates:

```text
B_v(D_k) = [P(v=1 | D_k), P(toggle_v | D_k)]
```

Then it removes the node's own average behavior across probes:

```text
mu_v      = mean_k B_v(D_k)
R_v(D_k) = B_v(D_k) - mu_v
S_v      = concat_k R_v(D_k)
```

`S_v` is the workload response signature used to supervise `h_b`. It is not a
model input feature.

## What Changed From The Earlier Version

The earlier version simulated each stochastic workload once. That made part of
the label depend on hidden random conditions that the model does not observe,
such as sampled PI probabilities, group latent sequences, and mixed-workload
perturbations.

The current version keeps the same `B -> mu -> R -> S` label structure, but
stabilizes stochastic workload probes by averaging multiple realizations before
constructing `R` and `S`:

```text
B_avg_v(D_k) = mean_m B_v(D_k^(m))
```

This makes the label better aligned with the intended training setup:

```text
(G, v) -> h_b(v) -> S_v
```

The seed strategy was also changed from shard-index dependent seeds to stable
hash seeds:

```text
blake2b64(global_seed, circuit_name, workload_name, realization_index)
```

Therefore the same circuit receives the same label regardless of shard
boundaries.

## Workloads

The default setup uses eight workload probes:

1. `uniform_p05`
2. `low_bias_p02`
3. `high_bias_p08`
4. `random_beta_05_05`
5. `low_toggle_markov`
6. `high_toggle_markov`
7. `group_correlation`
8. `mixed_bias_temporal_corr`

Fixed probes are simulated once per circuit:

```text
uniform_p05
low_bias_p02
high_bias_p08
low_toggle_markov
high_toggle_markov
```

Stochastic probes are estimated by averaging `num_realizations` independent
realizations. The default full configuration uses `num_realizations = 16`:

```text
random_beta_05_05
group_correlation
mixed_bias_temporal_corr
```

The `mixed_bias_temporal_corr` workload is now generated as a biased Markov base
sequence mixed with group latent behavior and small noise, instead of using a
large Bernoulli(0.5) XOR perturbation that can erase the original PI bias.

## Output Schema

`behavior_labels.npz` contains:

- `labels`: circuit-name keyed dictionary
- `meta`: configuration and schema metadata

For each circuit:

- `B`: `[K, num_nodes, 2]`, columns are logic-1 probability and toggle probability
- `R`: `[K, num_nodes, 2]`, residual behavior `B - mean_k(B)`
- `S`: `[num_nodes, K * 2]`, flattened response signature and main `h_b`
  supervision target
- `mu`: `[num_nodes, 2]`
- `sensitivity`: `[num_nodes]`, alias of `sensitivity_var`
- `sensitivity_var`: `[num_nodes]`, `sum_behavior_dim mean_k R_v(D_k)^2`
- `sensitivity_sum`: `[num_nodes]`, old-style `sum_k ||R_v(D_k)||_2^2` for comparison
- `num_pis`: scalar

For the current default labels, `K = 8` and `behavior_dim = 2`, so:

```text
S: [num_nodes, 16]
```

## Run

Debug run:

```bash
PYTHONPATH=simProject python simProject/scripts/build_behavior_labels.py \
  --config simProject/configs/debug.json
```

Full single-process run:

```bash
PYTHONPATH=simProject python simProject/scripts/build_behavior_labels.py \
  --config simProject/configs/default.json
```

Override the stochastic workload averaging count:

```bash
PYTHONPATH=simProject python simProject/scripts/build_behavior_labels.py \
  --config simProject/configs/default.json \
  --num-realizations 16
```

Reduce progress logging frequency:

```bash
PYTHONPATH=simProject python simProject/scripts/build_behavior_labels.py \
  --config simProject/configs/default.json \
  --log-every 100
```

## Reproduce Current Full Labels

The current official labels were generated in four shards and then merged. This
is safer than a single long run because each shard is written to disk
independently.

Generate shards:

```bash
PYTHONPATH=simProject python simProject/scripts/build_behavior_labels.py \
  --config simProject/configs/default.json \
  --start 0 --end 2500 --log-every 100 \
  --output simProject/outputs/behavior_labels_m16_shard_00000_02500.npz \
  --meta-output simProject/outputs/label_meta_m16_shard_00000_02500.json

PYTHONPATH=simProject python simProject/scripts/build_behavior_labels.py \
  --config simProject/configs/default.json \
  --start 2500 --end 5000 --log-every 100 \
  --output simProject/outputs/behavior_labels_m16_shard_02500_05000.npz \
  --meta-output simProject/outputs/label_meta_m16_shard_02500_05000.json

PYTHONPATH=simProject python simProject/scripts/build_behavior_labels.py \
  --config simProject/configs/default.json \
  --start 5000 --end 7500 --log-every 100 \
  --output simProject/outputs/behavior_labels_m16_shard_05000_07500.npz \
  --meta-output simProject/outputs/label_meta_m16_shard_05000_07500.json

PYTHONPATH=simProject python simProject/scripts/build_behavior_labels.py \
  --config simProject/configs/default.json \
  --start 7500 --end 10182 --log-every 100 \
  --output simProject/outputs/behavior_labels_m16_shard_07500_10182.npz \
  --meta-output simProject/outputs/label_meta_m16_shard_07500_10182.json
```

Merge shards:

```bash
PYTHONPATH=simProject python simProject/scripts/merge_behavior_shards.py \
  --shards \
    simProject/outputs/behavior_labels_m16_shard_00000_02500.npz \
    simProject/outputs/behavior_labels_m16_shard_02500_05000.npz \
    simProject/outputs/behavior_labels_m16_shard_05000_07500.npz \
    simProject/outputs/behavior_labels_m16_shard_07500_10182.npz \
  --output simProject/outputs/behavior_labels.npz \
  --meta-output simProject/outputs/label_meta.json
```

## Stability Check

Check whether the realization count is stable enough on a subset:

```bash
PYTHONPATH=simProject python simProject/scripts/check_realization_stability.py \
  --config simProject/configs/default.json \
  --limit 20 \
  --num-vectors 4096 \
  --realizations 8,16 \
  --output-json /tmp/realization_stability_4096_m8_m16.json
```

In the pilot check, `M=8` already produced high response-signature cosine
similarity to `M=16`, but the top-sensitive-node overlap was still low enough
that the default full-label setting was changed to `M=16`.

## Notes

DeepGate2 `labels.npz` can still be used later on the training machine for the
original DeepGate2 losses or for hard-negative mining with truth-table
similarity. It is not used to generate `S_v`.
